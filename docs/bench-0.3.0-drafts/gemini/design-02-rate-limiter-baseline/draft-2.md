## Rate limiting

The API employs a dual-layered rate limiting strategy to maintain service availability while protecting against abuse from both authenticated and anonymous traffic. Unauthenticated requests are restricted based on the originating IP address using a sliding window algorithm, ensuring fair resource distribution for public consumers. Conversely, authenticated traffic is governed by per-token quotas, which provide significantly higher throughput and more granular control for registered developers. All limits are transparently communicated via standard HTTP response headers—specifically `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset`—enabling clients to implement proactive backoff strategies before encountering `429 Too Many Requests` errors. Developers who encounter persistent throttling or believe their legitimate usage patterns require higher capacity can initiate a formal quota review through the developer portal’s support ticket system. This appeal path ensures that scale-up requirements are handled manually by the engineering team, allowing for flexibility without compromising the system's baseline security posture or overall performance stability.
