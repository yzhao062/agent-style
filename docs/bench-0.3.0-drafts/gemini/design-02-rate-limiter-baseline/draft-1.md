## Rate limiting

To ensure system stability and fair resource distribution, the API employs a tiered rate-limiting strategy tailored to different traffic profiles. Anonymous requests are throttled based on the originating IPv4 or IPv6 address, providing a baseline protection against automated scraping and distributed denial-of-service attempts. In contrast, authenticated traffic is governed by per-token quotas associated with specific API keys, allowing for significantly higher throughput and personalized service levels for registered developers. All API responses include standard HTTP headers—specifically `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset`—to provide real-time visibility into current consumption and cooling periods. When a limit is exceeded, the server returns a `429 Too Many Requests` status code with a descriptive body. Developers who believe their limits have been applied incorrectly or who require temporary quota increases for high-impact application launches can submit a formal appeal via the Developer Support Portal. Each request is manually reviewed to balance individual developer needs with global infrastructure health, ensuring a responsive, secure, and scalable ecosystem for our entire user base.
