## Rate Limiting

The API uses a dual-tiered rate limiting strategy to maintain service availability and prevent resource exhaustion. Per-IP limits apply to anonymous traffic, while authenticated requests operate under more generous per-token quotas. This system prioritizes known developers while protecting overall performance from unauthenticated callers. Current usage and remaining capacity data reach clients through standard HTTP response headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset`. If requesters exceed their allotted quotas, the server returns a `429 Too Many Requests` status code. Developers who believe their traffic patterns justify higher thresholds or who encounter persistent false positives must submit a formal limit increase request through the developer portal. Engineering personnel review these appeals to verify the use case and adjust policies for specific accounts. Such a tiered approach balances open accessibility with the security requirements of a high-traffic public interface.
