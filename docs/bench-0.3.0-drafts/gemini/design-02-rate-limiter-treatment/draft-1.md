## Rate Limiting

The system maintains availability by enforcing distinct rate limits for authenticated and anonymous traffic. Anonymous requests rely on a per-IP throttling mechanism to prevent resource exhaustion from unauthenticated sources. Authenticated users instead use a per-token quota, which provides higher throughput and more granular control over individual client activity. Clients track their consumption via standard `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` HTTP response headers. When an application exceeds these thresholds, the server returns a `429 Too Many Requests` status code containing a payload that specifies the mandatory cooling period. Engineering teams who believe their traffic patterns warrant an exception or encounter erroneous blocks must submit a formal request through the developer portal ticket system. Support personnel review these appeals to grant custom limit increases for verified, high-volume integrations that demonstrate legitimate architectural needs.
