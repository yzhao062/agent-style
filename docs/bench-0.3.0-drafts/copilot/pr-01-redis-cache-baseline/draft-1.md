This change replaces the in-process LRU session cache with Redis so session state is shared across instances, survives app restarts, and removes cache inconsistency during horizontal scaling. The rollout plan is to ship Redis support behind a configuration flag, enable dual-read/Redis-write in staging, then gradually flip production traffic by environment while monitoring cache hit rate, latency, and Redis error volume before removing the old LRU path. The biggest trade-off is that session lookups now depend on a networked external service, which adds operational complexity and can increase tail latency compared with local memory access. In return, we get consistent cross-node session behavior, better scale-out characteristics, and a clearer path for failover and capacity management.
