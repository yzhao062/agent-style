## Rate limiting

The public REST API applies layered rate limits to protect shared capacity while keeping common integrations predictable. Anonymous requests are limited per source IP, with lower burst and sustained thresholds to reduce abuse and accidental hot looping from unauthenticated clients. Authenticated requests are primarily limited per access token, with higher quotas that track the caller’s intended usage, while a secondary per-IP safeguard remains in place to contain misconfigured clients and token sharing from a single origin. When a client approaches or exceeds a limit, the API returns standard HTTP 429 responses and includes clear limit metadata in response headers, including the applicable limit, remaining requests, reset time, and the dimension enforced (IP or token). Error bodies also include a machine-readable reason code for easier client handling. Developers who believe they are being limited incorrectly can open a support ticket through the developer portal with request IDs, timestamps, and token details for review and, if warranted, limit correction or exemption.
