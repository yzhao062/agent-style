## Rate limiting

The public REST API enforces layered rate limits to protect availability while supporting both anonymous and authenticated use cases. Anonymous requests are limited per source IP, with a lower baseline intended for browser, preview, and light integration traffic; authenticated requests are evaluated against both the caller’s IP and API token, with the token-based limit serving as the primary quota and the IP-based limit acting as an abuse safeguard. This allows multiple legitimate users behind shared networks to operate normally while still constraining concentrated bursts from a single origin. Clients are informed of enforcement through standard HTTP `429 Too Many Requests` responses and accompanying headers that expose the applicable limit, remaining quota, reset time, and, when available, a `Retry-After` value to guide backoff behavior. Documentation and developer dashboards will explain limit tiers and common causes of throttling. If a developer believes traffic is being limited incorrectly, they may open a support appeal with request timestamps, token identifier, and observed response headers so the platform team can review logs, validate classification, and adjust policy or exemptions where warranted.
