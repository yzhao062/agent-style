## Rate limiting

The API enforces two complementary limits to protect shared capacity while keeping common client patterns simple. Anonymous requests are governed by a per-IP budget, tuned for bursty public traffic and intended to absorb basic browsing, evaluation, and low-volume integrations without requiring credentials. Authenticated requests are governed primarily by a per-token budget, which isolates tenants and prevents one application from consuming another application's quota; per-IP safeguards still apply as a secondary control to reduce abuse from shared or rotating networks. Clients receive limit details on every response through standard rate-limit headers that report the applicable ceiling, remaining requests, and reset time, and the service returns HTTP 429 with a machine-readable error body when a limit is exceeded. If a developer believes limits are being applied incorrectly, they may open a support ticket through the developer portal with the token ID, approximate timestamps, response headers, and request IDs so the platform team can review logs, correct classification errors, and grant relief where warranted.
