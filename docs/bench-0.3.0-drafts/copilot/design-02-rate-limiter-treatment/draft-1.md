## Rate limiting

The public REST API applies layered rate limits to protect shared capacity while remaining predictable for integrators. Anonymous requests are limited per source IP, with a lower burst and sustained quota to reduce abuse and protect unauthenticated endpoints. Authenticated requests are evaluated per API token and also subject to a broader per-IP ceiling, which prevents a single network origin from bypassing controls by rotating tokens. When a request is throttled, the API returns `429 Too Many Requests` and includes standard rate-limit headers that report the applicable limit, remaining quota, and reset time, plus a response body that identifies whether the IP-based or token-based policy triggered the denial. Documentation will publish default thresholds and note that stricter safeguards may apply during incident response. Developers who believe they are being limited incorrectly may open a support ticket with request timestamps, token identifier, and originating IPs; the platform team will review telemetry and adjust or exempt the client when warranted.
