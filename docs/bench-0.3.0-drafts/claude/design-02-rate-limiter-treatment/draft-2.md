## Rate limiting

The API enforces two concurrent buckets: a per-IP bucket of 60 requests per minute for anonymous traffic, and a per-token bucket sized to the caller's plan tier (default: 600 per minute, burst 1200) for authenticated traffic. Both buckets use a token-bucket algorithm refilled each second, counted independently so that a single authenticated client does not exhaust the shared anonymous pool behind its NAT. Every response carries `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers, and rejected requests return HTTP 429 with a `Retry-After` value in seconds and a machine-readable error code identifying which bucket tripped. Developers who believe a limit fires incorrectly, such as a batch job misclassified under the anonymous pool or a tier upgrade not yet propagated, may file a ticket at ratelimit-appeal@, attaching the request ID returned in `X-Request-Id`; on-call engineers review appeals within one business day and can issue a temporary override while the root cause is diagnosed.
