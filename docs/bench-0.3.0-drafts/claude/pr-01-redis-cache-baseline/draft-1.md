Switches the session cache from an in-process LRU to Redis so sessions are shared across app instances, eliminating sticky-session requirements and the cold-cache hit after each deploy. Rollout is gated behind a `SESSION_CACHE_BACKEND` flag: we'll enable Redis in staging for 48 hours, then ramp production at 5% → 25% → 100% over three days, with the LRU path retained as an instant fallback via flag flip. Monitoring additions cover Redis hit rate, p99 GET latency, and connection-pool saturation, with alerts wired to the on-call rotation. The biggest trade-off is that session reads now depend on a network round-trip and Redis availability — adding ~1–2 ms p50 latency and making Redis a hard runtime dependency — in exchange for horizontal scalability and cross-instance consistency.
