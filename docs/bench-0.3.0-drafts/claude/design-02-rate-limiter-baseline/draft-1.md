## Rate limiting

The API enforces two complementary rate limit tiers to balance open access with abuse prevention. Anonymous traffic is limited per source IP at 60 requests per minute and 1,000 per hour, while authenticated requests are metered per API token at 600 requests per minute and 10,000 per hour, with higher ceilings available for enterprise tokens. Limits are enforced by a centralized token-bucket service fronting all edge nodes, ensuring consistent accounting across regions. Every response carries `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers; when a caller exceeds its budget, the gateway returns HTTP 429 with a `Retry-After` header and a JSON error body describing the exhausted bucket. Developers who believe a limit was applied incorrectly—for example, due to shared NAT egress or a misattributed token—can file an appeal through the developer portal's "Rate limit review" form, which routes to the API platform team for a response within two business days and supports temporary uplifts while under investigation.
