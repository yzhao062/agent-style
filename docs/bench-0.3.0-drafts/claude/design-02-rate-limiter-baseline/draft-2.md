## Rate limiting

The API enforces a two-tier token bucket scheme to protect shared capacity while keeping legitimate traffic unimpeded. Anonymous requests are limited per source IP at 60 requests per minute with a burst of 120, tracked in a Redis-backed sliding window keyed on the client address (or the leftmost `X-Forwarded-For` hop behind our trusted proxies). Authenticated traffic is limited per API token at 600 requests per minute, with higher tiers configurable for enterprise plans; the token limit supersedes the IP limit once a valid credential is presented. Every response carries `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers, and a `429 Too Many Requests` response additionally includes `Retry-After` alongside a machine-readable JSON body identifying which bucket was exhausted. Developers who believe a limit was applied incorrectly — shared-NAT false positives, misclassified tokens, or post-incident quota restoration — may file an appeal through the developer portal's "Rate limit review" form, which routes to the API platform team with a 2-business-day SLA.
