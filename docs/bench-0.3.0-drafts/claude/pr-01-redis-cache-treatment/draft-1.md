This change replaces the in-process LRU session cache with Redis, so every application instance reads and writes a single source of truth for session state. The shared store removes the cross-pod inconsistency inherent to per-instance caches and lets us scale horizontally without cache-affinity routing. Rollout proceeds in three stages: deploy behind a `SESSION_CACHE=redis` flag to staging, mirror writes in production for 48 hours while comparing hit rates, then cut reads over once the error budget holds. The main trade-off is latency, since every session lookup now crosses the network and adds roughly 1-2 ms per request in exchange for correctness.
