Schema Drift Watch compares the database schema your services expect against the schema your production database actually has. It reads ORM models, migration files, and raw SQL queries from each service repository, then queries the live database for its current column definitions, indexes, and constraints. When a service expects a `users.email_verified` boolean that production has dropped, or when production gained a `payment_method` column that every service's models miss, the tool flags the gap before the next deploy.

The product targets backend teams running five or more services against at least one shared database. At that scale, the schema is too large for any single engineer to hold in their head, and the database becomes a quiet integration surface where one team's migration silently breaks another team's read path. A reviewer approving a pull request in the orders service lacks an efficient way to check whether the inventory service still queries the column being dropped, and a column added by the analytics team rarely shows up in the type definitions used by the checkout team.

Schema Drift Watch closes that gap. It runs on every pull request and on a schedule against production, then produces a per-service report showing which columns each service reads, writes, or assumes exists. When a migration is proposed, the tool lists every other service that depends on the affected columns, so the author and reviewer see the blast radius before merging rather than after a 3 a.m. page.

Install it once per repository, point it at a read replica, and trade the silent incidents that follow schema changes for a checked contract between code and database.
