Schema Drift Watch compares the database schema assumed by your service code against the schema actually running in production. It parses migration files, ORM models, and raw SQL queries, then cross-references them with live introspection of the target database. When a column gets added, dropped, renamed, or retyped outside the normal migration path, Schema Drift Watch flags the mismatch before the next deploy cycle.

The tool is built for backend teams running five or more services against a shared database, where ownership is distributed and no single engineer tracks every migration merged this quarter. A payments team might add a nullable column to the orders table for a new feature while a reporting team still assumes the column does not exist and breaks on its next aggregate query. Schema Drift Watch catches this class of silent incident before it reaches production traffic.

Anyone who has debugged a Monday morning outage traced back to Friday's migration recognizes the pain. Code review covers the diff in the repository, but the diff does not capture what a column addition implies for three downstream services that query the same table through different ORM layers. Tests pass because staging ran against a schema with the column already in place, while production still ran the older version. By the time the stack trace surfaces in the logs, the on-call engineer has inherited a problem that five services quietly created together.

Schema Drift Watch runs in CI, on a cron, or as a pre-deploy gate. Structured diff reports ship to Slack, GitHub, or any webhook, scoped by service owner. For teams past the point where one person can hold the schema in their head, the tool turns invisible coupling into explicit signals.
