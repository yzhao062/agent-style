## Rate limiting

The public REST API enforces rate limits at two levels to protect service reliability while preserving fair access. Anonymous requests are limited per source IP address, with stricter defaults because no durable caller identity is available. Authenticated requests are limited primarily per API token, so each developer or application has an isolated quota regardless of shared infrastructure, and a secondary per-IP limit remains in place to detect abusive fan-out or compromised clients. When either limit is exceeded, the API returns `429 Too Many Requests` with a machine-readable error code and standard rate-limit headers showing the active limit, remaining requests, reset time, and recommended retry interval. Clients should honor `Retry-After` and use exponential backoff with jitter for repeated failures. Developers who believe they are being limited incorrectly can open an appeal through the developer support portal, including token ID, affected endpoints, timestamps, request IDs, and expected traffic pattern; support can review logs, correct misclassification, or grant a higher quota where justified.
