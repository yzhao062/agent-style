## Rate limiting

The public REST API enforces layered quotas to protect availability while keeping anonymous access useful. Every request is counted against a per-IP bucket, with stricter limits for unauthenticated traffic and burst allowances that refill continuously; requests carrying a valid API token are also counted against a per-token bucket tied to the developer account or application. A request is rejected only when either applicable bucket is exhausted, so authenticated clients should send tokens consistently and avoid sharing tokens across unrelated workloads. Responses include `RateLimit-Limit`, `RateLimit-Remaining`, and `RateLimit-Reset` headers for the active constraint, plus `Retry-After` on `429` responses; documentation shows the current default limits and any endpoint-specific reductions. Clients should treat these headers as authoritative and implement exponential backoff with jitter. Developers who believe they are being limited incorrectly can file an appeal from the developer dashboard, including request IDs, timestamps, source IP ranges, token IDs, and expected traffic pattern; support will review logs, correct misclassification, or approve higher quotas.
