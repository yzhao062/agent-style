## Rate limiting

The API applies rate limits with two token buckets: a per-IP bucket for all traffic and a per-token bucket for authenticated traffic. Anonymous requests consume only the per-IP bucket. Authenticated requests must pass both checks: the token bucket protects each developer allocation, and the IP bucket limits abusive concentration behind one source address. Default anonymous limits are lower than authenticated limits, and trusted tokens may receive higher quotas after review. Every response includes `RateLimit-Limit`, `RateLimit-Remaining`, and `RateLimit-Reset` headers for the effective constraint nearest exhaustion. When a request is rejected, the API returns `429 Too Many Requests`, a `Retry-After` header, and a JSON error body with the limit scope, reset time, and request ID. Developers who believe throttling is incorrect can open an appeal in the developer portal with request IDs, timestamps, token identifier, expected traffic pattern, and business impact; support reviews logs and either corrects misclassification or explains the applicable quota.
