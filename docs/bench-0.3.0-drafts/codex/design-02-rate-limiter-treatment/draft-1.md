## Rate limiting

Requests are limited on two independent dimensions: a per-IP budget for all callers and a per-token budget for authenticated callers. Anonymous traffic uses only the IP budget, so bursts from one network cannot crowd out other clients. Authenticated traffic consumes the token budget for the API key or OAuth client and also remains subject to a higher per-IP safety cap, which limits abuse from shared infrastructure without punishing normal token use. When either budget is exhausted, the API returns `429 Too Many Requests` with a JSON error that names the limiting scope and a `Retry-After` value. Every response includes `RateLimit-Limit`, `RateLimit-Remaining`, and `RateLimit-Reset` headers for the active scope, so clients can back off before failure. Developers who believe a limit is wrong should open a support ticket from the developer dashboard with the request ID, token ID, timestamps, source IPs, and expected traffic pattern; support can correct classification errors or approve a quota increase.
