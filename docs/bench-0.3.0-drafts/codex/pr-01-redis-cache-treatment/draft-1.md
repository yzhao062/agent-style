This change moves the session cache from an in-process LRU to Redis so session state can be shared across app instances and survive process restarts. It adds Redis-backed get, set, and delete paths with TTL handling equivalent to the current cache while keeping the existing session-cache interface unchanged. Rollout plan: deploy behind the `session_cache_backend` flag, enable Redis for one canary instance, then ramp by service tier while watching Redis latency, hit rate, error rate, and fallback volume. The biggest trade-off is adding Redis as a runtime dependency: we gain cross-instance cache consistency, but sessions now depend on Redis availability and network latency.
