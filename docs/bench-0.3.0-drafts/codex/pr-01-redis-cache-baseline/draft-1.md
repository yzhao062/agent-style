This change switches the session cache from an in-process LRU to Redis so session state is shared consistently across app instances and survives process restarts. The rollout plan is to enable Redis-backed caching behind configuration in staging first, monitor latency/error rates/cache hit rates, then progressively enable it for production traffic with a fast rollback to the existing LRU path if needed. The single biggest trade-off is adding an external network dependency on Redis, which improves correctness in distributed deployments but can introduce cache latency or availability impact if Redis is degraded. Operational dashboards and alerts should be watched closely during rollout to confirm Redis capacity, connection health, and session behavior under real traffic.
